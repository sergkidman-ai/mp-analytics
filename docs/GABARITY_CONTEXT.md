## 1. СУТЬ ПРОБЛЕМЫ

Обе площадки берут логистику по объёму, вычисленному из ДхШхВ, которые МЫ сами вводим в карточку:
- **WB (FBS):** логистика ~пропорциональна объёму в литрах (`volume_l = Д×Ш×В / 1000`).
- **Ozon:** логистика по **объёмному весу** = `объём_см³ / 5000` (кг).

Раздутый/незаполненный короб в карточке → систематическая переплата за каждую отгрузку.
Исправляется коррекцией ДхШхВ в карточке (Контент WB / атрибуты Ozon) — пересчёт тарифа
происходит у площадки автоматически со следующих поставок.

## 2. СДЕЛАНО (код + данные в БД)

### Модель данных (миграции)
- `006_wb_cards_dims.sql` — в `wb_cards`: `length_cm/width_cm/height_cm, weight_kg, volume_l, dims_valid`.
- `036_supplier_dims.sql` — таблица `supplier_dims` (габариты ЗАКУПОЧНОЙ упаковки поставщиков);
  ключ связки `article` = `products.article`; объём/вес приведены к `volume_l`/`weight_kg`.
- `037_ozon_dims.sql` — таблица `ozon_dims` (задекларированные габариты карточек Ozon);
  `depth/width/height_mm, weight_g, volume_l`; связка с продажами по `sku`/`offer_id`.

### Коллекторы
- `collectors/wb_card_content.py` / `collectors/wb.py` — габариты карточек WB → `wb_cards`.
- `collectors/supplier_dims.py` — РУЧНЫЕ выгрузки прайсов поставщиков (`incoming/size.zip → incoming/gab/`).
  У каждого поставщика свой формат единиц (приводим к см/кг/л):
  - **Изи (T2)** — «Размеры мм» одной ячейкой `Д*Ш*В`, вес кг;
  - **Cactus** — Длина/Высота/Ширина (мм) по колонкам, вес брутто кг;
  - **Сакура** — Д/Ш/В в МЕТРАХ + объём м³, вес кг (листы Лазерная/Струйная/Матричная);
  - **Профилайн** — только Вес и Объём (м³), БЕЗ ДхШхВ и штрихкода.
- `collectors/ozon_dims.py` — габариты карточек Ozon из `GET /v4/product/info/attributes`
  (ДхШхВ мм + вес г) → литры + объёмный вес.

### Аналитика переплаты
- **WB** (`docs/wb_logistics_overpay.md` + `.csv`, 406 строк): регрессия факт-логистики на объём
  карточки дала эмпирический тариф **логистика/шт ≈ 46 + 29,8 × объём_л** (1 л→76 ₽, 5 л→195 ₽,
  11 л→374 ₽). Каждый лишний литр ≈ **30 ₽/шт**.
- **Ozon** (`docs/ozon_dims_overpay.csv`, 159 SKU): переплата по объёмному весу, декл. объём vs реальный.
- Коммиты: `feat(fin): габариты поставщиков → переплата по логистике WB` и
  `83c43e9 feat(fin): габариты Ozon из API → точная переплата по объёму` (оба 2026-07-15).

### Дашборд (web/app.py)
- Коммиты `31fa716` (2 маржи, габариты/плотность, ФБО-стоимость), `7788e64` (в убыточных —
  рекомендация «поправить габариты» вместо цены при подозрительном объёме).
- Флаг плотности `кг/л` на продаваемых карточках, пороги **DENS_LOW=0.05, DENS_HIGH=0.7**
  (медиана ≈0.14, p05≈0.04, p95≈0.61):
  - `< 0.05` = «крупн./лёгкий» → раздут объём, лишние литры логистики;
  - `> 0.7` = «тяжёлый/мелкий» → занижены габариты (часто опечатка, напр. длина 1 см);
  - `dims_valid=false` = «невалидные» (WB пометил).
- Блок «Подозрительные габариты» + подсказка «N убыточных позиций — из-за габаритов, а не цены».
- `analyze_return_plan.py` тоже использует `volume_l`/`dims_valid` (реальный объём карточки WB) в
  плане «не в минус».

## 4. ГИПОТЕЗЫ И ОГОВОРКИ (почему это оценка СВЕРХУ)

- Габариты поставщика = закупочная упаковка ОДНОГО картриджа — прокси реального размера. Наш
  ОТГРУЗОЧНЫЙ короб может быть больше (несколько штук, наполнитель) → фактическая экономия ниже.
- Покрыты только 4 поставщика (~30% продаж WB). Остальное появится по мере добавления прайсов.
- WB-тариф выведен ЭМПИРИЧЕСКИ из наших данных (регрессия), не из офиц. сетки WB — годится для
  ПРИОРИТИЗАЦИИ, не для точного биллинга.
- Ozon: та же болезнь и та же оговорка (декларация vs реальный размер отгрузки).

## 5. ОТКРЫТЫЕ ВОПРОСЫ

1. **Применены ли коррекции карточек?** Аналитика есть (топ-CSV), но исправлены ли реально ДхШхВ
   в Контенте WB / атрибутах Ozon — не зафиксировано. Нужен статус «правки внесены / эффект пошёл».
2. **Реальный отгрузочный короб** vs закупочная упаковка — надо подтвердить фактические размеры хотя
   бы по топу (иначе экономия завышена). Источник истины — собственные обмеры/фото упаковки.
3. **Официальная сетка тарифов WB/Ozon** — заменить эмпирическую регрессию на точную формулу для
   расчёта экономии до рубля (сейчас только приоритизация).
4. **Покрытие поставщиков WB** — как дотянуть матчинг с 30% до большей доли (недостающие прайсы).
5. **Профилайн** отдаёт только вес+объём (нет ДхШхВ, нет штрихкода) — как связывать/использовать.
6. **Мониторинг** — карточки со временем снова «расплываются» (новые SKU на дефолтах); нужен
   регулярный отчёт-детектор, а не разовый разбор.
