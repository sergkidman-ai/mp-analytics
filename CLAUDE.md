# CLAUDE.md — mp-analytics («Пульт бизнеса»)

BI-система маркетплейс-аналитики для продавца картриджей «Цифровой квадрат»: забрать по API
данные всех площадок, свести в одну БД, считать реальную экономику каждого SKU.

**Перед любой задачей читать `ARCHITECTURE.md`** — это карта проекта (видение, принципы,
дорожная карта). Этот файл — только быстрый операционный ориентир.

## Это отдельный проект

Живёт **только** в `/opt/mp-analytics` (свой git, свои сессии Claude Code, своя память).
НЕ трогать соседние проекты: `/opt/sokol-server` (Сокол — анализ ТЗ), `/opt/tz-analyzer-src`,
`/var/www/sokol`. Кросс-правок между проектами нет.

## Инфраструктура (уже развёрнута)

- **БД:** PostgreSQL 16 в Docker (контейнер `mp-postgres`), `127.0.0.1:5433`, база `mp_analytics`,
  том `/opt/mp-analytics/pgdata` (переживает рестарт). Доступ — `DATABASE_URL` из `.env`.
- **Python:** venv в `/opt/mp-analytics/venv` (Python 3.10). Запуск скриптов — `./venv/bin/python ...`.
- **Веб-дашборд:** `uvicorn web.app:app` на `127.0.0.1:8090`, systemd-юнит `web/mp-dashboard.service`.
- **git:** репозиторий здесь, ветка `main`.

## Структура

- `collectors/` — сбор по API (moysklad, wb, ozon*, yandex*, *_ads, *_funnel, wb_jam). Сырьё в raw-слой.
- `core/db.py` — подключение к БД и хелперы: `query` / `execute` / `upsert` (идемпотентный) / `apply_sql_file`.
- `reports/` — витрины расчётов (`margin_by_sku.py`, `margin_ozon_sku.py`, `ozon_expenses.py`).
- `migrations/NNN_*.sql` — схема БД по шагам (применять через `core.db.apply_sql_file`).
- `web/` — FastAPI-дашборд (`app.py`, `static/`).
- `run_daily.py` — оркестратор cron: скользящее окно (текущий+прошлый месяц), все аккаунты.
- `analyze_*.py`, `*_plan.py`, `promo_*.py` — разовые аналитические скрипты.
- `docs/` — рабочие отчёты/анализы.

## Аккаунты

WB: `wb_acc1`, `wb_acc2`. Ozon: `oz_acc1` (Премиум-Про, есть отзывы/рейтинг), `oz_acc2` (Дисквэр).
Различаются полем `account`; площадка — полем `platform`.

## Два потока работы (домены) — читай свой бриф первым

Работа разнесена на параллельные сессии-домены. Каждая сессия первым делом читает свой бриф:
- **fin** — Финансы/экономика: себест, COGS, маржа SKU, выручка/расходы, P&L. Бриф: `docs/BRIEF_FIN.md`,
  оркестратор `run_daily.py`, миграции **0xx**.
- **mkt** — Маркетинг/видимость: Джем (позиции/воронка/запросы), реклама/ДРР, ABC, просадки.
  Бриф: `docs/BRIEF_MKT.md`, оркестратор `run_marketing.py`, миграции **1xx**.

**Граница:** `margin_by_sku` пишет ТОЛЬКО fin; mkt её читает (read-only). Параллельные сессии — каждая
в своём git worktree и ветке `fin/*` / `mkt/*`. **Флаг чужой территории** активен через git-хук
(`tools/territory_guard.py`, `git config core.hooksPath tools/hooks`): коммит правок чужих файлов
блокируется. Самопроверка: `python3 tools/territory_guard.py --status`. Подробности — в брифах.

## Команды

```bash

./venv/bin/python run_daily.py        # полный прогон сбора + пересборка витрин

```

## Правила

1. **МойСклад — источник правды** по товарам и себестоимости. Артикул — единый ключ связи МС↔площадки.
2. **Сырьё отдельно от расчётов:** raw-слой (JSONB) хранит ответы API как есть; любой расчёт
   пересобирается из сырья без повторного обращения к API.
3. **Идемпотентность:** повторный сбор за период не плодит дублей — `upsert` по натуральному ключу.
4. **Секреты только в `.env`** (gitignored). Никогда не хардкодить и не выводить ключи в чат.
5. **Себест для анализа** брать из витрины `margin_by_sku` (ключ `article` = nm_id), НЕ наивным
   джойном `external_code → cost_seb` (дырявый путь). Себест конкретной отгрузки — `report/stock`
   на ДАТУ отгрузки, не усреднённый. Подробности — в памяти проекта.
6. Каждый шаг сверять с реальностью (контроль: nmID 216421567 за май 2026 → чистая **3741.77 ₽**).
   Обновлено 2026-07-16: было ≈3909 ₽; коммит `9031962` исправил знак строк «Возврат» и добавил вычет
   баллов лояльности (`cashback_amount`) в удержания — чистая честно снизилась, сверено с ЛК ВБ Δ<0.03%.
7. **Сторно COGS возвратов WB** (`ms_return_cogs` + `margin_by_sku`). Когда покупатель возвращает
   товар и он идёт в ПРОДАВАЕМЫЙ сток, себест исходной отгрузки возвращается из расходов (иначе при
   перепродаже спишется дважды). Политика:
   - **A. Период:** сторно в МЕСЯЦ возврата (по строке «Возврат» `raw_wb_report`, = месяц денежного реверса).
   - **B. Оценка:** себест/шт = `cogs/qty` исходной отгрузки из кэша `ms_demand_cogs` × штук возврата.
     Мост к отгрузке — `salesreturn.demand.name` (= demand.name = WB assembly_id); НЕ по `salesreturn.name`.
   - **D. Гейт по складу:** сторнируем для ВСЕХ складов, КРОМЕ **«Брак»** (Звездный/Дисквер/Кантемировская
     = сток); «Брак» = дефект, себест остаётся расходом. Гейт кэшируется в `ms_return_cogs.sellable`.
   - Покрытие частичное (только возвраты с assembly_id и себестом в кэше) — остальное честный остаток,
     сторно не завышается. Пустая `ms_return_cogs` = сторно не применяется (безопасная деградация).
   - **Ozon** (`ms_return_cogs` platform=ozon + `margin_ozon_sku`): та же логика. **A.** месяц возврата =
     месяц возвратной операции `raw_ozon_transaction`. **B.** себест/шт = `cpu_global` (всеисторический
     matched-себест SKU, т.к. продажа обычно в другом месяце), иначе групповой fallback по offer_id.
     **D. Гейт по складу:** сторнируем сток (Звездный/Дисквер/Кантемировская), НЕ сторнируем **{Брак, Озон}**
     («Озон» = товар, удержанный на стороне Озона, не наш сток). **Мост:** `salesreturn.demand.name` =
     `posting_number`; ключ = первые 2 сегмента (norm_posting). **Кол-во:** возврат даёт ДВЕ операции —
     денежную (`ClientReturnAgentOperation`, accr<0) и товарную (`OperationReturnGoodsFBSofRMS`); берём
     **MAX** штук на SKU (одна физед., не сумма), кап по МС-бюджету sellable на постинг (делит смешанные
     Брак/сток). Сторно бакетируется по месяцу возвратной операции → в этом месяце COGS SKU может уйти в
     минус (это корректная кредит-проводка себеста, зеркалящая денежный реверс возврата).
   - **Яндекс.Маркет** (`yandex_monthly._ms_cogs_monthly`): COGS **order-based** (Σ `cost_seb`×qty по
     позициям МС-customerorder за месяц заказа), поэтому сторно считается ВНУТРИ функции, БЕЗ таблицы
     `ms_return_cogs`. **A.** вычитаем в **МЕСЯЦ ЗАКАЗА** (где себест и учтён; `salesreturn.demand.name`
     = `customerorder.name` = номер заказа ЯМ) — не в месяц возврата, иначе реверс попадёт мимо (и
     переусторнирует возвраты по заказам вне окна). **B.** себест возврата = Σ `cost_seb`×qty по
     позициям самого `salesreturn`, кап по себесту заказа. **D.** сторнируем сток
     (Звездный/Кантемировская), НЕ **«Брак»**. Возвраты по заказам вне окна `since` пропускаем (их
     себест не в окне). Себест-минусов по месяцу не бывает (вычет ≤ вклад заказа в этот месяц).
8. **Сплит строки «Продажи» Ozon** (`collectors/ozon_realization.py` + дашборд). ЛК раскладывает
   «Продажи» на **Выручка + Баллы за скидки + Программы партнёров**. Единственный точный источник —
   **отчёт о реализации** `/v2/finance/realization` (помесячный, `raw_ozon_realization`), НЕ постинги:
   `financial_data.customer_price/price` этот сплит не воспроизводят (на oz_acc1 совпало случайно, на
   oz_acc2 расходилось на 19%). Маппинг `delivery_commission` (проверено Δ=0 оба юрлица, июнь-2026):
   Выручка=`amount`, Баллы=`bonus`, Партнёры=`bank_coinvestment`+`pick_up_point_coinvestment`+`stars`.
   Отчёт формируется **после закрытия месяца** — текущий месяц отдаёт 404 (`fetch`→None, не ошибка).
   Верхняя строка «Продажи» дашборда остаётся accruals-транзакциями (Δ к реализации 0.0–0.3%); сплит —
   отдельным блоком «по отчёту о реализации» (вариант A, не трогает маржу/COGS).
9. **При старте сессии ПЕРВЫМ делом читать `docs/HANDOFF.md`** — актуальное состояние передачи между
   сессиями (что сделано, активные ветки, открытые вопросы, следующий шаг), затем свой бриф `docs/BRIEF_*.md`.
10. **В конце значимого блока работы обновлять `docs/HANDOFF.md`** — текущее состояние / сделанное /
    активные ветки / открытые вопросы / следующий шаг. Это предохранитель от рассинхрона и потери контекста
    при переполнении сессии (см. память про 32MB и устаревший эталон).

## Делегирование Codex

Codex (OpenAI) подключён как MCP-инструмент (`mcp__codex__codex`, `mcp__codex__codex-reply`,
user scope). Выход к OpenAI идёт через forward-proxy на релее — прокси уже прописан в env
регистрации MCP, трогать/дублировать его не нужно. Правила делегирования:

1. **Анализ/ревью — только `sandbox: read-only`.** Файл `.env` не читать, секреты не выводить
   (правило 4 распространяется и на Codex).
2. **Правки кода — `sandbox: workspace-write`, всегда в ОТДЕЛЬНОЙ git-ветке** (`fin/*` или `mkt/*`
   по домену). Одна правка = одна ветка = точечный diff.
3. **Git делает Claude, не Codex.** Sandbox Codex блокирует запись в `.git`, поэтому ветку/коммит
   создаёт оркестратор (Claude). Коммитить ТОЛЬКО целевые файлы правки; не подхватывать чужие
   незакоммиченные изменения в рабочем дереве.
4. **В `main` не мержить без явного «ок» пользователя.** Границы доменов fin/mkt и `territory_guard`
   обязательны и для Codex.
5. **Финансовые правки проверять цифрами:** после патча — прогон отчёта на реальном месяце и дельта
   в рублях до/после. Эталон — цифры площадок; подгонять запрещено, остаточные расхождения объяснять.
6. `approval-policy: never` допустим только внутри этих sandbox-рамок (read-only / workspace-write).
