# Разведка МойСклада — что доступно по API (для плана сбора)

Дата: 2026-06-13. Read-only обследование боевого аккаунта (бренд «Цифровой квадрат»).
Источник правды по товарам/закупкам/операциям. Продажи-деньги (комиссии/логистика) — на стороне маркетплейсов.

---

## 1. Идентификаторы товара (ключевое)
У карточки `entity/product`:
- `externalCode` — **группа-модель** (напр. `0002` = Q2612A), заполнен 100%. На неё «обрезается» маркетплейс-SKU (`00024` → `0002`). **Точка стыковки с витриной.**
- `code` — `0002` + аббревиатура бренда (`0002sk`, `0002ep`, `0002oem`…). Заполнен ~89%. **НЕ уникален** (в группе 0002: `0002cs×5`, `0002sp×4`, `0002ep×3`).
- `article` — артикул производителя (`SAQ2612A`). ~82%.
- `id` (ms_id) — **единственный уникальный ключ карточки** → выбран как PK таблицы `products`.

Каталог: **44 284 товара** (архивных 0 — аккаунт не архивирует, вместо этого атрибут «Стоп лист»).

## 2. Кастомные атрибуты карточки (в разделе 6.1 их не было)
- **`Длина, см.` / `Ширина, см.` / `Высота, см.`** — габариты (разреженные; основной объём — из карточек WB).
- `weight` (top-level) — вес, заполнен (0.785 кг…). `volume` top-level = 0.
- **`Название WB`** — имя WB-листинга (одинаково у карточек одной группы) → **кандидат на ключ связи МС↔WB**.
- **`Связь`** — ещё один маркетплейс-ключ.
- `Бренд`, `Код поставщика`, `EAN/UPC`, `Цвет`, `Модель`, `Ресурс печати`, `Список принтеров`, `Стоп лист`(bool), ячейки складов.
- `pathName` — категория (`Картриджи/Картриджи HP`).
Сырьё (`raw_moysklad_product.payload`, JSONB) хранит ВСЕ атрибуты — не теряем.

## 4. Склады (9), юрлица (3), каналы (2)
- **Склады:** `Озон`, `Москва для ФБО`, `СПБ для ФБО`, **`Брак`**, `Транзит`, `Кантемировская`, `Звездный`, `Дисквер`, `Удаленный`. Остатки/движения — по каждому. ФБО-склады отделяют маркетплейс-логистику; **брак считается по складу «Брак» + документам `loss`**.
- **Юрлица:** `ООО "ЦИФРОВОЙ КВАДРАТ"`, `BeYindi`, `ООО "ДИСКВЭР"`.
- **Каналы продаж:** только `СДЕК`, `kartridge.org` — аккаунты НЕ кодируют.

## 5. Статусы (на заказах покупателей)
`customerorder` несёт богатый статус-цикл, включая возвраты/претензии/отмены:
`Доставлен`, `В пути`, `Ожидает вручения`, `Претензия`, `Обмен`, `Претензия решена возврат ДС`,
`Возврат в пути на наш склад`, `Возврат на склад МП`, `Отменен - невыкуп / дорого / оригинал / наличие / недозвон` и др.
Отгрузки (`demand`) собственного статуса не имеют.

## 6. Разделение аккаунтов — ЧАСТИЧНО
- **Площадка** различается агентом заказа: `Покупатель ВБ`, `Покупатель Озон`, `Покупатель Маркет`.
- **account-уровень (ВБ-аккаунт 1 vs 2) в МС по агенту НЕ виден** — он на уровне площадки.
- Вывод: 2 аккаунта одной площадки разделять **со стороны API маркетплейса** (токен = аккаунт);
  МС даёт агрегат по площадке (возможно, по юрлицу/складу — сопоставить, когда заведём токены).

## 7. Что отдаёт отгрузка (demand) построчно
`expand=positions.assortment,agent,organization`:
- какая **карточка** ушла (`code`/`externalCode`), **qty**, **наша цена**;
- **площадка** (agent), **юрлицо** (organization), **склад** (store), номер заказа маркетплейса, `moment`.
- фильтр по `moment >= <с момента>` → инкрементальный сбор «в течение дня».

## 8. Пример среза: продажи группы 0002 за месяц (2026-05-14 … 06-13)
Просмотрено 5 510 отгрузок. **Продано 0002: 28 шт, ≈ 28 694 ₽ (по нашей цене).**
- Площадки: ВБ 26 шт / Озон 1 / Маркет 1.
- Юрлицо: всё «Цифровой квадрат».
- Реально отгружались только 2 карточки из 33: `0002oem` ×20 (ср. 982 ₽), `0002ct` ×8 (ср. 1132 ₽).

**Вывод:** «чья себестоимость применилась» решается фактом из отгрузки (какой бренд ушёл), а не «минимумом по группе». Площадка/юрлицо/поставщик считаются однозначно из МС.

## 9. Нюансы / на будущее
- demand = ОТГРУЖЕНО; возвраты (`salesreturn`) и невыкупы (статусы) уменьшат нетто — учитывать отдельно.
- Цена в demand — наша, НЕ выплата площадки и не себестоимость.
- Себестоимость группы для маржи — MIN по карточкам с `buy_price > 0` (в группе есть пустышки с 0).
- Производительность: серверного фильтра demand по товару нет → грузить demand инкрементально по `moment` в raw, агрегаты считать в SQL.
- МС закрывает: штуки, поставщик, площадка, юрлицо, статусы, возвраты, брак, склады. Маркетплейсы — деньги (комиссия/логистика/выплата/реклама/воронка/позиции).

## 10. Природа сверки МС↔WB и лаг (КЛЮЧЕВОЕ для расчётов)

Сверка мая (Цифровой квадрат + Покупатель ВБ) vs WB-финотчёт `wb_acc1`:
- **ДЕНЬГИ сходятся:** МС 3 948 900 ₽ vs WB 3 895 332 ₽ (+1.4%, по нашей цене).
- **ШТУКИ расходятся:** МС 2180 vs WB 1612 — это by design, не ошибка.

**Почему штуки/заказы НЕ сойдутся (и не должны):**
- **Набор цветных:** на WB — 1 юнит (1 nm_id), в МС — N карточек-компонентов (разные цвета).
  Деньги = цена набора = Σ компонентов. (Двойных одинаковых нет — есть наборы цветных.)
- **Мультизаказ:** несколько штук в одном заказе WB дробит на N «заказов» (чтобы списать
  логистику N раз); в МС это 1 заказ.

**Лаг (измерено на мае):**
- `rr_dt` (дата в отчёте) = `sale_dt` (выкуп): медиана 0 дней — отчёт НЕ задерживается.
- `sale_dt − order_dt` (выкуп − заказ): **медиана 6 дней, средняя 6.7** — это доставка/выкуп.
- МС фиксирует на ОТГРУЗКЕ (`moment`, раньше), WB — на ВЫКУПЕ (`sale_dt`, +6–7 дн).
- → примерный лаг МС→WB ≈ **6–7 дней**; финотчёт недельный → свежая неделя неполная.

**Правила для витрин:**
1. Сверка и доверие — по ДЕНЬГАМ и по ЗАКАЗУ, НЕ «штуки=штуки».
2. COGS WB-продажи = Σ себестоимостей реально отгруженных компонентов из МС (особенно для наборов).
3. WB агрегировать по `sale_dt`; тянуть с перекрытием ~2 недели; последние ~7–10 дней = неполные (completeness-флаг).
4. Связка WB↔МС: через номер заказа (WB report ↔ MS demand) и/или nm_id→vendorCode→атрибут «Название WB».
