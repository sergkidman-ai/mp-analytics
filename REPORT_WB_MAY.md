## 5. Сверка МС ↔ WB за май (Цифровой квадрат + Покупатель ВБ)
| | МС (по факту) | WB-финотчёт |
|---|---|---|
| Деньги (наша цена) | 3 948 900 ₽ | 3 895 332 ₽ → **+1.4%, сходятся** |
| Штуки | 2 180 | 1 612 → расходятся by design |

**Штуки не сойдутся никогда (и не должны):**
- Набор цветных: WB = 1 юнит, МС = N карточек-компонентов. Деньги = цена набора = Σ компонентов.
- Мультизаказ: WB дробит на N «заказов» (ради логистики), в МС — 1 заказ.
→ Сверка и расчёты — по **деньгам** и по **заказу**, COGS — по **физическим компонентам** из МС.

## 6. Остатки на складах WB (= FBO по данным WB) и почему 27% без COGS
То, что лежит на складах WB, WB считает как FBO. Читаем из их API остатков:
- доступно **516 шт**, quantityFull **864**, в пути к клиенту **290**, **возвраты в пути 58**, **725 nmID**;
- ориентир. стоимость остатка **≈1 690 686 ₽**; раскидано по складам WB (Екатеринбург, Коледино, Краснодар…).
- Продажи с этих остатков WB отгружает сам → **в МС нет поштучной отгрузки** → нет `assembly_id`-связки → 27% без COGS.
- **В МС склады WB не заведены** — этот остаток в МС НЕ виден. COGS для FBW-продаж берём по товару (`nm→vendorCode→карточка МС→buy_price`).

## 7. Лаг (измерено)
- `rr_dt` (в отчёте) = `sale_dt` (выкуп): медиана 0 дней — отчёт не задерживается.
- `sale_dt − order_dt` (выкуп − заказ): **медиана 6 дней, средняя 6.7** — доставка/выкуп.
- МС фиксирует на ОТГРУЗКЕ (раньше), WB — на ВЫКУПЕ (+6–7 дн). Финотчёт недельный → свежая неделя неполная.
- Правило: WB агрегировать по `sale_dt`, тянуть с перекрытием ~2 недели, последние ~7–10 дней = неполные.
